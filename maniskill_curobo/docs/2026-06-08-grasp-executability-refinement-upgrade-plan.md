# 抓取可执行性修正模块后续升级计划

## 1. 文档目的

本文记录 ZeroGrasp 与 cuRobo 抓取系统后续的升级方向，供后续开发、实验设计和结果评审使用。

这不是当前必须立即完成的开发任务，也不代表文中所有阶段都要一次性实现。后续应按照实验结果逐步推进，避免在基础问题尚未确认时直接引入复杂模型。

## 2. 当前系统与问题

当前抓取链路大致为：

```text
ManiSkill 场景
    -> RGB、深度图和目标掩码
    -> ZeroGrasp 输出抓取位置、姿态、宽度和 depth
    -> 坐标系与 panda_hand/TCP 标定转换
    -> cuRobo 规划 pre-grasp、grasp 和 lift
    -> ManiSkill 执行并判断是否成功 lift
```

目前已经确认，ZeroGrasp 输出的抓取在几何上看起来合理，并不代表机械臂一定能够执行。问题主要分为两类：

1. **规划层失败**
   - pre-grasp 无法规划；
   - grasp 无法规划；
   - lift 无法规划；
   - 抓取姿态与桌面碰撞，cuRobo 找不到无碰撞 IK。

2. **物理抓取失败**
   - 夹爪没有包住物体；
   - 夹爪闭合时把物体推走；
   - 抓取深度太浅，抬升时滑落；
   - 抓取深度太深，轨迹无法规划或接近时撞到物体；
   - 抓住后 lift 路径不合适。

### 2.1 已有实验依据

在 `PickSingleYCB-v1` seed 1-200 的 zero residual 实验中：

- 33 个场景在 grasp 阶段规划失败；
- 保留桌面碰撞时，这 33 个目标全部找不到无碰撞 IK；
- 移除桌面碰撞、保留机器人自碰撞后，33/33 全部能够规划；
- 目标物体已从 cuRobo 障碍物中排除，因此失败源是夹爪或机械臂与桌面的碰撞，不是目标物体被误当成障碍；
- 这些失败中包含较多平放的刀、叉、勺、剪刀、铲子和螺丝刀。

在 full depth 实验中还发现：

- 将 ZeroGrasp depth 同时加到 pre-grasp 和 grasp，会让部分物体在接近阶段被撞动；
- 固定原始 pre-grasp，只对最终 grasp 应用 depth 后，接近过程更稳定；
- 对 grasp depth 使用 `100% -> 75% -> 50% -> 25% -> 0%` 自动回退，15 个已知退化场景全部恢复为成功 lift；
- 不同物体选择的有效 depth 比例不同，因此不能用一个固定 depth 阈值解决所有情况。

这些结果说明，zero 与 full depth 不应被看作两个互不相关的执行模式。它们都需要经过同一个“可执行性检查与修正模块”。

## 3. 总体目标

在不修改 ZeroGrasp 主模型输出含义的前提下，在 ZeroGrasp 与 cuRobo 之间增加一个统一的抓取可执行性修正模块。

该模块负责：

1. 优先检查原始抓取是否可以直接执行；
2. 原始抓取可行时保持不变；
3. 原始抓取不可行时，只做尽可能小的修正；
4. 同时考虑规划可行性与实际 lift 成功率；
5. 所有修正均经过 cuRobo 碰撞检查；
6. 保存每次尝试，为后续监督学习或强化学习提供数据。

目标链路为：

```text
ZeroGrasp top-k 抓取候选
    -> 原始候选可执行性检查
    -> 失败原因诊断
    -> 最小修正候选生成
    -> cuRobo 规划过滤与排序
    -> 执行抓取
    -> 保存规划和物理结果
```

## 4. 核心设计原则

### 4.1 必须保留零修正

修正模块不能默认改变所有 ZeroGrasp 输出。原始抓取能够成功规划时，零修正候选应拥有最高优先级。

这样可以防止出现：

```text
原始 ZeroGrasp 可以成功
    -> 修正模块强行改动
    -> 抓取反而失败
```

### 4.2 规划成功不等于抓取成功

cuRobo 只负责回答“机械臂能否无碰撞到达”，不能保证夹爪一定能稳定抓住物体。

因此评价指标必须同时包含：

- pre-grasp 是否规划成功；
- grasp 是否规划成功；
- lift 是否规划成功；
- 是否接触或包住物体；
- 是否成功 lift；
- 抬升后是否稳定保持；
- 是否在接近或闭合过程中明显推动物体。

### 4.3 不通过关闭碰撞检测解决问题

移除桌面碰撞只能用于诊断，不能作为正式抓取方案。

正式系统必须保留：

- 桌面碰撞；
- 机器人自碰撞；
- 工作空间限制；
- 必要的安全距离。

否则仿真中的“规划成功”无法迁移到真机。

### 4.4 修正应由小到大

修正候选按照与原始抓取的偏差从小到大尝试。只有较小修正无法解决问题时，才扩大搜索范围或切换 top-k 候选。

## 5. 第一阶段：确定性自动修正器

第一阶段不训练神经网络，先使用 cuRobo 作为可执行性判定器，建立稳定、可解释的自动修正流程。

### 5.1 输入

- ZeroGrasp top-k 抓取候选；
- 每个候选的位置、旋转、宽度、depth 和分数；
- 目标物体局部点云；
- 桌面平面或桌面碰撞体；
- 当前机械臂关节状态；
- cuRobo 场景碰撞模型。

### 5.2 修正变量

建议允许以下修正：

```text
位置：Δx、Δy、Δz
姿态：Δroll、Δpitch、Δyaw
抓取深度：depth scale
夹爪开口：Δwidth 或 width scale
候选选择：top-k candidate index
```

其中：

- `Δz` 优先表示沿桌面法向抬高，解决夹爪碰桌；
- depth scale 解决 full depth 过深或 zero depth 过浅；
- 小角度旋转解决指尖、手掌或腕部朝向桌面的问题；
- 横向修正主要用于改善包络位置，不应成为第一优先级；
- width 修正主要解决夹空、推物体和闭合位置不合适。

### 5.3 分级尝试顺序

建议采用分阶段搜索，避免一次性枚举所有组合。

#### 第一级：原始候选

对每个 ZeroGrasp 候选先尝试：

1. zero depth；
2. full depth；
3. 已有的 depth 自动回退比例。

pre-grasp 始终锚定在 zero-depth 抓取姿态之外，depth 只作用于最终 grasp。

#### 第二级：桌面避碰修正

当 cuRobo 判断最终 grasp 与桌面冲突时，优先尝试：

- 沿桌面法向抬高；
- 根据夹爪最低点与桌面的间距计算所需最小抬高量；
- 在最小抬高基础上增加少量安全余量；
- 必要时增加小角度倾斜，让手指离开桌面。

应优先使用几何计算得到最小修正，而不是盲目遍历大量固定高度。

#### 第三级：小范围位姿修正

当抬高后虽然可规划，但抓取几何明显变差时，再尝试：

- 局部横向平移；
- 绕抓取 approach axis 或 width axis 的小角度旋转；
- depth 与姿态联合微调；
- width 小范围调整。

#### 第四级：切换 top-k 候选

如果当前候选在合理修正范围内仍不可执行，应切换 ZeroGrasp 的下一个候选，而不是无限扩大 residual。

候选优先级可以综合：

- ZeroGrasp 原始分数；
- cuRobo 是否可规划；
- 与原始抓取的修正距离；
- 桌面与机器人碰撞余量；
- 轨迹长度和关节运动量；
- 预测的物理抓取稳定性。

### 5.4 候选评分

规划成功的候选不应只按“第一个成功”选择。后续可以使用如下代价排序：

```text
总成本 =
    位置修正成本
  + 姿态修正成本
  + depth 修正成本
  + width 修正成本
  + 轨迹长度成本
  + 低碰撞余量惩罚
  - ZeroGrasp 分数奖励
```

原则是优先选择：

- 改动小；
- 碰撞余量足够；
- 轨迹短；
- 仍然保持较高 ZeroGrasp 置信度的候选。

## 6. 第二阶段：完整记录自动修正数据

确定性修正器稳定后，应把每次尝试保存为训练数据。

### 6.1 每条记录至少包含

- 环境、seed、物体类别和物体实例；
- 原始局部点云；
- 原始 ZeroGrasp top-k 输出；
- 原始位置、姿态、width、depth 和分数；
- 尝试的 residual；
- 修正后的完整抓取姿态；
- pre、grasp、lift 各阶段规划结果；
- 失败阶段与 cuRobo 诊断；
- 是否碰桌、是否自碰撞、是否超工作空间；
- 实际执行是否完成；
- 物体接近前后、闭合前后、lift 前后的位置变化；
- 是否成功 lift；
- 是否掉落、夹空或推动物体；
- 成功视频路径和必要的失败视频路径。

### 6.2 标签应分层

不要只保存一个 `success/failure`。

建议至少拆成：

```text
planner_feasible
pre_plan_success
grasp_plan_success
lift_plan_success
contact_success
object_lifted
stable_lift
object_pushed
object_dropped
```

这样才能区分：

- 姿态根本不可执行；
- 姿态可执行但抓不住；
- 抓住但 lift 路径失败；
- 抬起后发生滑落。

## 7. 第三阶段：学习型修正头

当自动修正数据量和覆盖范围足够后，再训练一个修正头，减少在线搜索次数。

### 7.1 建议输入

- 抓取点附近的局部原始点云；
- ZeroGrasp 原始抓取位置与旋转；
- ZeroGrasp width、depth 和分数；
- 桌面平面在抓取坐标系中的表示；
- 当前抓取姿态到桌面的估计最小距离；
- 可选的机械臂状态或简单可达性特征。

局部点云应使用 PointNet、PointNet++ 或其他点云编码器直接学习，而不是只依赖少量手工统计量。

### 7.2 建议输出

```text
Δx、Δy、Δz
Δroll、Δpitch、Δyaw
depth scale
width scale
候选有效性或成功概率
```

如果同时预测多个修正方案，可以输出若干候选及其置信度，再交给 cuRobo 过滤。

### 7.3 推荐训练顺序

1. 使用自动修正器产生的成功 residual 做监督学习；
2. 使用成功和失败尝试训练候选价值或成功率模型；
3. 用 cuRobo 对模型输出继续做安全过滤；
4. 在独立验证 seed 上比较搜索次数、规划率和 lift 成功率；
5. 只有监督学习稳定后，再考虑强化学习。

### 7.4 学习头不是安全检查器

即使模型预测了一个看起来合理的 residual，也必须经过 cuRobo 检查。

学习头的作用是更快提出高质量候选，不是取代碰撞检测。

## 8. 第四阶段：强化学习微调

强化学习应解决“规划可行但物理抓取质量仍不够”的问题，不应承担基础碰撞安全职责。

### 8.1 观测

- 局部点云编码；
- ZeroGrasp 原始抓取；
- 当前 residual；
- 桌面平面和碰撞余量；
- 前一次尝试的规划结果；
- 接近、闭合和 lift 后的物体位移；
- 是否接触、推动、抬起或掉落。

### 8.2 动作

建议继续使用抓取参数 residual，而不是直接控制七个机械臂关节：

```text
[Δx, Δy, Δz, Δroll, Δpitch, Δyaw, depth/width adjustment]
```

机械臂关节轨迹仍由 cuRobo 生成。

### 8.3 奖励

奖励应区分规划与物理结果：

- 稳定 lift：最高正奖励；
- 短暂 lift 后掉落：低于稳定 lift；
- 有效接触但未 lift：小正奖励或中性反馈；
- 夹空：负奖励；
- 推动物体：负奖励；
- grasp/lift 规划失败：负奖励；
- 碰撞、越界或非法姿态：较强负奖励；
- residual 过大：小幅动作惩罚。

不建议让 `outside_workspace` 或单一规划失败惩罚完全支配奖励，否则策略可能只学会输出保守但无抓取意义的姿态。

## 9. 评测方案

### 9.1 数据划分

必须固定训练集、验证集和测试集：

- 训练 seed：用于收集和更新模型；
- 验证 seed：用于选参数和 checkpoint；
- 测试 seed：只用于最终比较。

后续正式评测默认至少使用 100 个固定测试 seed，并保证其中包含：

- 球体；
- 盒状物；
- 细长工具；
- 平放薄物体；
- 不规则物体；
- 高、矮、宽、窄等不同几何形态。

### 9.2 对照组

每次升级至少比较：

1. ZeroGrasp zero depth；
2. ZeroGrasp full depth；
3. depth 自动回退；
4. 确定性统一修正器；
5. 学习型修正头；
6. 学习型修正头加 cuRobo 安全过滤。

所有对照必须使用相同：

- seed；
- ZeroGrasp 候选；
- 场景碰撞模型；
- 工作空间；
- settle 时间；
- lift 判定；
- 视频与日志策略。

### 9.3 核心指标

- pre、grasp、lift 的规划成功率；
- 总执行完成率；
- lift 成功率；
- 稳定 lift 成功率；
- 原本成功但修正后失败的退化率；
- 原本失败但修正后成功的改善率；
- 平均尝试次数；
- 平均规划时间；
- 平均 residual 大小；
- 碰桌、推物体、夹空和掉落的比例；
- 未见物体与未见 seed 上的泛化能力。

## 10. 推荐实施顺序

### 阶段 A：完成规则修正闭环

- 统一 zero 与 depth 的入口；
- 保留零修正候选；
- 固定 zero-depth pre-grasp；
- 完善 depth 自动回退；
- 加入基于桌面间距的最小抬高；
- 加入小角度姿态避碰；
- 加入 top-k 候选切换；
- 建立统一结果日志。

### 阶段 B：扩大数据覆盖

- 在多种物体和姿态上运行自动修正；
- 保留成功与失败尝试；
- 清洗不稳定 reset、错误 ZeroGrasp 输出和场景导出异常；
- 建立训练、验证和测试划分。

### 阶段 C：训练监督修正头

- 使用自动搜索成功结果做行为克隆；
- 训练候选可执行性和 lift 成功率估计；
- 与规则修正器做固定测试集对比。

### 阶段 D：必要时加入强化学习

- 从监督模型 warm-start；
- 保留 cuRobo 安全过滤；
- 只优化物理抓取效果和搜索效率；
- 在独立测试 seed 上验证是否真正优于规则搜索。

## 11. 风险与注意事项

1. **抬高可避免碰桌，但可能使夹爪抓得过浅。**
   因此抬高与 depth、姿态需要联合考虑。

2. **规则搜索可能形成有限枚举。**
   第一阶段允许使用分级搜索验证规律，但长期目标是几何计算加学习候选，而不是无限增加固定动作模板。

3. **ZeroGrasp 输出可能本身明显错误。**
   residual 不应被允许无限大。偏差过大时应切换候选或判定该场景无有效抓取。

4. **仿真精确碰撞体与真机感知存在差距。**
   后续需要评估桌面平面估计误差、物体点云不完整和机器人标定误差，并保留合理安全余量。

5. **训练成功率不能直接当成模型测试性能。**
   在线探索期间的成功来自“采样加更新”，最终性能必须用冻结策略、确定性输出和未参与训练的测试 seed 衡量。

6. **不能只训练失败场景。**
   必须保留原始成功抓取及零 residual 数据，教会模型在不需要修改时保持原输出。

## 12. 最终预期

理想状态下，系统面对一个新的 ZeroGrasp 输出时能够：

1. 判断原始抓取是否可以直接执行；
2. 可以执行时不作多余修改；
3. 碰桌时自动产生最小避碰修正；
4. depth 不合适时自动选择合适比例；
5. 当前候选不可用时切换 top-k 候选；
6. 在所有候选执行前使用 cuRobo 做安全过滤；
7. 最终提高 lift 成功率，同时降低原本成功场景的退化率。

这套升级的核心不是让模型随意改变 ZeroGrasp，而是建立一个从“视觉上合理的抓取”到“机械臂能够安全执行并稳定抬起”的中间层。
